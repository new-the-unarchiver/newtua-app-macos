# Приложение для macOS переехало в свой репозиторий

Дата: 2026-07-23. Адресат: ассистент, который ведёт приложение.

Всё время своей жизни приложение лежало каталогом `apps/macos` внутри общего
репозитория `newtheunarchiver` и собиралось вместе с Rust-кодом, который лежал
рядом. Этого больше нет. Ниже — что случилось, что от этого сломалось, как
чинить и куда смотреть.

---

## 1. Что произошло

Общий репозиторий разделён на пятнадцать отдельных: движок, форматы, обёртки,
интерфейсы — каждый со своей историей, своей лицензией и своими выпусками.
Библиотеки опубликованы на crates.io, привязки для Python — на PyPI, пакет для
Swift — своим выпуском на GitHub.

Приложение вынесли последним, 2026-07-23. Внутри него **не изменён ни один
файл**: каталог `apps/macos` целиком, вместе с 39 коммитами своей истории,
поднят в корень нового репозитория.

Новый адрес: **https://github.com/new-the-unarchiver/newtua-app-macos**
Рабочая копия на диске: `/Users/trankov/proj/rust/newtua/newtua-app-macos`
Единственная ветка — `main`.

**Пути стали короче на префикс `apps/macos/`.** Было
`apps/macos/NewTheUnarchiver/…` — стало `NewTheUnarchiver/…`. Файлы те же,
корень репозитория другой.

**Старый общий репозиторий мёртв.** Его нельзя собирать, тестировать и указывать
в качестве источника зависимостей. Он остаётся на диске только как архив.

## 2. Кто теперь где

Всё это — организация `new-the-unarchiver` на GitHub, и всё это лежит на диске
соседними каталогами в `/Users/trankov/proj/rust/newtua/`.

| Что | Репозиторий | Было в общем репозитории |
|---|---|---|
| Движок распаковки | `newtua-core` (crates.io, `0.2.0`) | `crates/newtua-core/` |
| C ABI и заголовок `newtua.h` | `newtua-ffi` (crates.io, `0.1.0`) | `crates/newtua-ffi/` |
| **Пакет SwiftPM `Newtua`** | **`newtua-swift`** (выпуск `v0.1.0`) | `bindings/swift/` |
| Приложение | `newtua-app-macos` | `apps/macos/` |

Остальные одиннадцать — форматы, вынужденные ответвления чужих библиотек,
локализация, командная строка, привязки для Python — приложения не касаются.

## 3. Что сломалось

Три места. Все три — привязки к старому расположению файлов; ни одно не связано
с кодом самого приложения.

**1. Ссылка на пакет Swift в проекте Xcode.** В `project.pbxproj` записано:

```
XCLocalSwiftPackageReference "../../../bindings/swift"
```

Раньше это вело в `bindings/swift` общего репозитория. Теперь такого каталога
нет — ни на три уровня вверх, ни где-либо ещё в репозитории приложения.

**2. Фаза сборки Run Script.** Она запускает
`$REPO_ROOT/apps/macos/tools/build-newtua-xcframework.sh`, а тот, в свою очередь,
делает `cargo build -p newtua-ffi` в `$REPO_ROOT/crates/newtua-ffi` и кладёт
результат в `$REPO_ROOT/bindings/swift/Newtua.xcframework`. Ни `crates/`, ни
`bindings/` в новом репозитории нет.

**3. `tools/install-prealpha.sh`.** В нём строка

```bash
SRCROOT="$REPO_ROOT/apps/macos/NewTheUnarchiver"
```

Каталога `apps/macos` больше нет — должно стать `$REPO_ROOT/NewTheUnarchiver`.

## 4. Как чинить — и хорошая новость

**Код приложения править не нужно вообще.** Файл `Sources/Newtua/Newtua.swift`
в репозитории `newtua-swift` **побайтно совпадает** с тем, что лежал в
`bindings/swift`. Обёртка та же, имена те же, поведение то же. Меняется только
то, откуда Xcode берёт пакет.

Что делать:

1. **Убрать локальную ссылку на пакет** и добавить вместо неё удалённую:
   `https://github.com/new-the-unarchiver/newtua-swift`, правило версии —
   точная `v0.1.0`. Имя продукта, к которому подключаются цели, прежнее —
   `Newtua`. Он по-прежнему динамический, значит в Xcode остаётся выпадающий
   список «Embed», и один и тот же фреймворк встраивается в приложение один раз
   и разделяется с расширением Quick Look через `@rpath` — ровно как было.

2. **Убрать фазу Run Script целиком.** Она больше не нужна: готовый
   `Newtua.xcframework` прикреплён к выпуску `v0.1.0` пакета и закреплён
   контрольной суммой. SwiftPM скачает его сам. **Rust на машине сборки больше
   не требуется** — ни `cargo`, ни `~/.cargo/bin` в `PATH`.

3. **Поправить `tools/install-prealpha.sh`** — убрать `apps/macos/` из пути.

Заодно, если сборка приложения где-то ждала скрипт сборки фреймворка перед
`swift test` в `bindings/swift`, — этой связки тоже больше нет: тесты обёртки
живут в своём репозитории и гоняются там.

**Что от переезда стало лучше:**

| | Было (сборка из общего репозитория) | Стало (выпуск `v0.1.0`) |
|---|---|---|
| Архитектуры | только `arm64` | `arm64` **и** `x86_64` |
| Нижняя планка macOS у фреймворка | 26.0 | 14.0 |
| Нужен ли Rust для сборки | да | нет |

У приложения `MACOSX_DEPLOYMENT_TARGET = 26.5`, то есть планка фреймворка (14.0)
заведомо ниже — совместимость полная, и вдобавок появились машины на Intel.

## 5. Отставание по версии движка — знать, но не чинить самому

Готовый `Newtua.xcframework` в выпуске `v0.1.0` собран из `newtua-ffi 0.1.0`,
который зависит от `newtua-core 0.1.1`. Движок с тех пор вышел в `0.2.0`.
Набор поддерживаемых форматов от этого не изменился — в `0.2.0` добавлен
публичный вызов определения кодировки имён, а не новые форматы, — так что
приложение ничего не теряет.

Обновление, когда оно понадобится, идёт цепочкой и **делается не в приложении**:
поднять зависимость в `newtua-ffi` → выпустить `newtua-ffi` → пересобрать
XCFramework → выпустить `newtua-swift` новой версией с новой контрольной суммой
→ и только потом приложение поднимает у себя правило версии пакета.

Важно: **прикреплённый к выпуску архив нельзя пересобирать поверх**. Контрольная
сумма закрепляет ровно этот файл, а повторная сборка даёт другой. Новая
сборка — только новая версия выпуска.

## 6. Что переписать в своих документах

Приложение везде описывает себя как часть общего репозитория. Эти места теперь
врут:

- **`CLAUDE.md`** — таблица путей (`bindings/swift/`, `crates/newtua-ffi/`,
  `crates/newtua-core/`, `apps/macos/tools/…`); раздел про сборку XCFramework
  скриптом; фраза «Корень репо — через `git rev-parse --show-toplevel`»
  (сам приём верный, но указывает уже на другой корень); строка «Править
  `crates/`, `Cargo.toml`, `Cargo.lock` из macOS-агента — handoff» — этих
  каталогов просто нет, запрет стал бессмысленным, а вместо него нужен порядок
  обращения к владельцу движка.
- **`NewTheUnarchiver/ARCHITECTURE.md`** — схема
  `crates/newtua-core → crates/newtua-ffi → bindings/swift`.
- **`.claude/skills/add-macos-format/SKILL.md`** — весь разбор «посмотри
  `crates/newtua-core/src/archive.rs`, `decompress.rs`». Сами файлы никуда не
  делись, но лежат в соседнем репозитории: на диске это `../newtua-core`, в
  сети — `github.com/new-the-unarchiver/newtua-core`. Добавление формата теперь
  выглядит так: формат появляется в движке, движок выходит на crates.io,
  дальше цепочка из раздела 5, и только потом приложение объявляет новый тип
  документа. Часть работы, которая касается приложения — UTI, `Info.plist`,
  Quick Look, — не изменилась.

## 7. Где смотреть документацию

- **Пакет Swift:** `newtua-swift` — `README.md` и `Package.swift` (в нём
  подробные пояснения, почему продукт динамический и как устроен двоичный
  таргет). На диске: `../newtua-swift`.
- **C ABI:** `newtua-ffi` — `README.md` и заголовок `include/newtua.h`,
  порождаемый cbindgen. На диске: `../newtua-ffi`.
- **Движок и список форматов:** `newtua-core` — `README.md`. На диске:
  `../newtua-core`.
- **Организация целиком:** https://github.com/new-the-unarchiver

Разбор того, в каком порядке распространяются выпуски между репозиториями,
лежит в закрытом репозитории процесса и наружу не выдаётся; всё, что нужно
приложению, изложено в разделе 5.

## 8. Границы

- **Движок, C ABI и обёртка Swift — чужая территория.** Раньше это выражалось
  правилом «`crates/` только на чтение»; теперь это просто другие репозитории.
  Нужна правка в C ABI или в обёртке — это запрос владельцу, а не коммит.
- **Приложение — твоя территория целиком**, включая свои документы и свой скрипт
  установки.
- Ветка в новом репозитории одна — `main`.
