# newtua — macOS GUI (Claude Code brief)

You are building the **macOS GUI** for `newtua`, a Rust archive **extractor** — a
modern rewrite of *The Unarchiver*. This file is the technical contract. The
human will give you the product/UX direction and screenshots of the original
app's screens; your job is to implement them faithfully **on top of the existing
Rust engine**, respecting the nuances below.

---

## 0. Prime directive

**Do not implement any archive logic in Swift.** All detection, listing,
decompression, password handling, encoding detection, path-safety, symlink and
permission/mtime restoration already live in the Rust engine and are exposed
through a Swift package. The GUI is **pure UI + orchestration**. If you find
yourself parsing archive bytes, computing extraction paths, or sanitizing
filenames in Swift — stop; that belongs in the engine.

This app **only extracts and lists**. It never creates archives.

---

## 1. Repository geography (paths are relative to the repo root)

This Xcode project is `apps/macos/NewTheUnarchiver/NewTheUnarchiver.xcodeproj`
(Xcode wraps a new project in a folder named after it — that nesting is normal).
So `SRCROOT` = `apps/macos/NewTheUnarchiver`, and the **repo root is three levels
up**. Don't hardcode `../../..`; find it with `git rev-parse --show-toplevel`
(used in the build script below) so paths survive any reorganization. The pieces
you depend on (relative to the repo root):

- `bindings/swift/` — a **local Swift Package** named `Newtua` (the product is
  `Newtua`). This is your dependency. It wraps the C ABI in idiomatic Swift.
- `crates/newtua-ffi/` — the Rust C-ABI crate. It produces:
  - the static library `target/debug/libnewtua_ffi.a` (after `cargo build`),
  - the C header `crates/newtua-ffi/include/newtua.h` (generated by cbindgen;
    the Swift package's `Sources/CNewtua/newtua.h` is a **symlink** to it).
- `crates/newtua-core/` — the engine. You will not touch it from Swift.
- `target/debug/` — where the linker finds `libnewtua_ffi.a`.

**You may extend `bindings/swift` (the Newtua package) when the Swift API needs
to be friendlier — see §3.** You should rarely need to touch the Rust crates; if
you do (e.g. to expose a missing flag), see §12.

## 2. How the app links the engine (read this carefully — it's the #1 setup trap)

The `Newtua` package's `Package.swift` links the Rust static library with
`unsafeFlags`: it computes an **absolute** path to `<repo>/target/debug` from the
manifest's own location and emits `-L<that> -lnewtua_ffi`, plus the native
libraries the Rust archive pulls in:

```
-framework CoreFoundation  -lc++  -lpthread  -llzma  -liconv  -lbz2
```

(`-lSystem -lc -lm` are implicit on macOS.) These are declared in the package, so
they propagate to your app automatically — **do not re-add them** in the Xcode
target unless you hit a missing symbol.

Consequences and traps:

1. **Add the package as a LOCAL package** (Xcode ▸ File ▸ Add Package
   Dependencies ▸ Add Local… ▸ choose `../../bindings/swift`). Xcode permits
   `unsafeFlags` only for local/root packages — a remote/versioned dependency
   with `unsafeFlags` is **rejected**. Local is fine.
2. **The static lib must exist before you build.** Add a **Run Script build
   phase** to the app target, ordered *before* "Compile Sources", that builds it:
   ```sh
   set -euo pipefail
   export PATH="$HOME/.cargo/bin:/opt/homebrew/bin:/usr/local/bin:$PATH"
   cd "$(git -C "$SRCROOT" rev-parse --show-toplevel)"  # repo root, wherever the .xcodeproj sits
   cargo build -p newtua-ffi  # writes target/debug/libnewtua_ffi.a + regenerates the header
   ```
   Xcode's build environment does **not** include `~/.cargo/bin` on `PATH` — the
   explicit `export` above is mandatory or `cargo` is "command not found".
   Uncheck "Based on dependency analysis" so it always runs.
3. **The `-L` path is hardcoded to `target/debug`.** It does *not* switch to
   `target/release` for an Xcode Release build. For development this is fine
   (a debug Rust lib links into a Release app — it's just unoptimized). For a
   real shipped build, see §11 (universal lib / XCFramework).
4. **The lib is large** (~100 MB debug, statically includes `libunrar` C++, 7z,
   lzma, bz2). First link is slow; that's expected.

## 3. The Newtua Swift API (and its rough edges — fix these first)

Public surface (`import Newtua`):

```swift
func version() -> String

struct NewtuaError: Error, CustomStringConvertible { let code: NtuaStatus; let message: String }
struct Entry  { let path: String; let kind: String /* "file"|"dir"|"symlink" */
                let size: UInt64; let isEncrypted: Bool; let mode: UInt32?; let mtime: Int64? }
struct ExtractReport { let extracted: UInt64; let failed: UInt64; let aborted: Bool }

final class Archive {
    init(path: String, password: String? = nil, encoding: String? = nil) throws
    var count: Int
    func entries() -> [Entry]
    func entry(at index: Int) -> Entry?
    func read(_ index: Int) throws -> Data
    @discardableResult
    func extract(to dest: String, selection: [Int]? = nil,
                 wrapper: Bool = true, strict: Bool = false, preserve: Bool = true,
                 progress: ((NtuaProgress) -> Bool)? = nil) throws -> ExtractReport
}
```

**Rough edges you should clean up before building UI on top (priority order):**

1. **`NewtuaError.code` is `NtuaStatus`, and `progress` takes `NtuaProgress` —
   both are C types from the `CNewtua` module, which the package does NOT
   re-export.** As shipped, your app can't name `NtuaStatus`/`NtuaProgress`
   without also `import CNewtua`. Fix it in the package, choosing one:
   - *Minimal:* add `@_exported import CNewtua` in `Sources/Newtua/Newtua.swift`.
   - *Better (do this):* replace the C types in the public API with Swift-native
     ones — a `public enum ErrorCode { case io, unknownFormat, unsupported,
     encrypted, wrongPassword, corrupt, pathTraversal, missingVolume,
     invalidIndex, nullArg, utf8, panic }` mapped from `NtuaStatus`, and a
     `public struct Progress { let index: Int; let path: String?; let
     bytesWritten: UInt64; let entrySize: UInt64; let started: Bool; let
     finished: Bool }` built inside `extract`'s trampoline (decode
     `NtuaProgress.path` with `String(cString:)` only when non-null). Then the
     app never imports `CNewtua`.
2. **`progress` is synchronous and fires on the extraction thread**, not the main
   thread (see §4).
3. **There is no async API.** Add `async` wrappers (or call from a background
   task) — see §4. Don't make the package itself spawn threads.

After any change to the package, run `cd bindings/swift && swift test` (it needs
`cargo build -p newtua-ffi` first). If you change the C ABI, the header is
regenerated by that cargo build and the symlink picks it up automatically.

## 4. Concurrency (get this right or you'll get hangs / data races)

- **`Archive` is NOT thread-safe.** It owns an opaque pointer to a single-threaded
  Rust reader. Use one `Archive` from **one** thread/queue at a time. Never call
  `read`/`extract`/`entries` on the same `Archive` concurrently.
- **`extract` (and `read` of a large entry) blocks.** Run it off the main actor —
  e.g. `Task.detached` or a dedicated serial `DispatchQueue`. Keep the UI
  responsive.
- **The `progress` closure is invoked on the extraction thread.** Do not touch
  SwiftUI state from inside it directly — hop to the main actor
  (`await MainActor.run { … }` / `DispatchQueue.main.async`) to update progress
  bars. Keep the closure cheap; it's called frequently (per entry start, per
  byte chunk, per entry done — see `started`/`finished`/`bytesWritten`).
- **Cancellation** is cooperative: return `false` from the `progress` closure to
  abort. Wire your Cancel button to a flag the closure reads (e.g. an
  `actor`-guarded bool or `Atomic`). On abort, `ExtractReport.aborted == true`.

A good shape: one serial queue ("engine queue") per opened archive; all engine
calls go there; progress hops to `@MainActor` for UI.

## 5. Memory & safety contract (the wrapper handles it — don't fight it)

- `Archive` frees its handle in `deinit` (`ntua_free`). Just let it deallocate;
  don't keep raw pointers.
- `Entry` values are **copies** (Swift `String`/scalars) — safe to keep after the
  `Archive` is gone.
- `read` returns a fresh `Data`; the underlying buffer is freed for you.
- **No Rust panic crosses the boundary** — every C entry point catches unwinds
  and returns an error code. So a corrupt archive surfaces as a thrown
  `NewtuaError`, never a crash. (A 412 GB-allocation OOM class was already fixed
  in the engine and is fuzz-guarded; you don't need to defend against it.)

## 6. Errors & the password flow (important UX nuance)

`NewtuaError.code` (a `NtuaStatus`) tells you *what* went wrong; `.message` is a
localized human string (English/Russian, from the engine's i18n). Map codes to UI:

- `Encrypted` / `WrongPassword` → prompt for a password.
- `UnknownFormat` → "unsupported or not an archive".
- `Corrupt` → "damaged archive".
- `PathTraversal` → only under `strict: true`; otherwise the engine skips the
  unsafe entry and continues.

**Password is set at `open`, not at `extract`.** The engine reads it when the
archive is opened. So the flow for an encrypted archive is:

1. `try Archive(path:)` (no password). If it throws `.Encrypted`/`.WrongPassword`,
2. prompt the user, then **open a NEW `Archive(path:, password:)`** and proceed.

You cannot "add a password later" to an already-open `Archive`. (`extract` has no
password parameter by design — that was intentional cleanup.)

## 7. Filenames & encoding

- `Entry.path` is already decoded to UTF-8 (the engine auto-detects legacy
  charsets across all names at once). Display it directly.
- Offer an **encoding override** (like the CLI's `--encoding`): pass `encoding:`
  to `Archive(path:encoding:)`, e.g. `"shift_jis"`, `"cp866"`, `"ibm866"`. Use
  WHATWG/`encoding_rs` labels. Changing it requires re-opening the archive.

## 8. Behaviors to mirror exactly (parity with The Unarchiver / our CLI+TUI)

- **"Create enclosing folder"** ⇔ the `wrapper` flag. When `wrapper: true`, if the
  archive's contents do **not** already share a single common top-level folder,
  the engine wraps them in a folder named after the archive (its file stem). If
  they already have one common root, nothing extra is added. Don't reimplement
  this — just expose the toggle.
- **Destination folder** — default "next to the archive"; let the user pick.
- **macOS sidecar metadata** (`._*`, `.DS_Store`, `__MACOSX/`) is **skipped by
  default** during extraction. NOTE: the C ABI's `ntua_extract` currently hardwires
  this (no toggle). If the GUI wants a "keep macOS metadata" option, you must add
  a flag to the C ABI + Swift wrapper (see §12). Don't fake it in Swift.
- **Symlinks, permissions, mtimes** are restored by the engine when
  `preserve: true` (default). Nothing to do in Swift.

Our CLI/TUI feature set, for reference (the GUI should cover the meaningful
subset the screenshots imply): browse the entry tree and tick a selection;
options = destination, password, encoding, "create folder" (wrapper), strict;
live progress with speed/ETA and cancel; a final summary with any failures.

## 9. macOS app concerns the engine does NOT handle (these are yours)

- **Drag & drop** of archive files onto the window/Dock (the signature TUA UX),
  plus **"Open With"** / double-click: declare archive UTIs in
  `CFBundleDocumentTypes` / `Exported`+`Imported Type Identifiers`
  (`public.zip-archive`, `org.7-zip.7-zip-archive`, `com.rarlab.rar-archive`,
  `public.tar-archive`, gzip/bzip2/xz, etc.) and handle the open-file events.
- **App Sandbox & entitlements** (decide with the human; required for Mac App
  Store, recommended otherwise): a sandboxed app needs
  `com.apple.security.files.user-selected.read-write` to write to a user-chosen
  destination (via `fileImporter`/`NSOpenPanel`). Files dropped/opened by the
  user come with read access. To **remember** a destination across launches you
  need **security-scoped bookmarks** (`startAccessingSecurityScopedResource`).
  The engine just calls `std::fs` — it has no knowledge of sandbox scopes, so the
  app must open the destination's scope **before** calling `extract` and close it
  after.
- **Hardened Runtime + notarization** for distribution outside the App Store.
  Static-linking our own Rust code is fine; no special exception needed. (See §11.)
- **`read(_:)` loads a whole entry into memory** — fine for "Quick Look"/preview
  of a selected file, but don't preview multi-GB entries. There is no streaming
  read in v1.
- **`entries()` copies the whole listing** each call — cache the array in your
  view model; don't call it in a render loop.

## 10. Don'ts

- Don't reimplement extraction/detection/path logic in Swift (§0).
- Don't block the main thread with `extract`/`read` (§4).
- Don't share one `Archive` across threads (§4).
- Don't free C pointers yourself or hold `Entry` C pointers — use the wrapper's
  Swift values.
- Don't add the package as a *remote* dependency (unsafeFlags will be rejected, §2).
- Don't bundle or shell out to `rar`/`unrar`/`7z`/`tar` binaries — everything is
  in-process in the static lib. (On macOS the Homebrew `rar`/`unrar` are
  Gatekeeper-blocked anyway.)
- Don't `cargo update` the `unrar` crate or touch `vendor/unrar-0.5.8` — it's a
  pinned, patched vendor copy; changing it reintroduces a multi-volume crash.

## 11. Distribution (when you get there — not day one)

- For a shippable, cargo-free build, produce a **universal static lib** and link
  that instead of `target/debug`:
  ```sh
  rustup target add aarch64-apple-darwin x86_64-apple-darwin
  cargo build -p newtua-ffi --release --target aarch64-apple-darwin
  cargo build -p newtua-ffi --release --target x86_64-apple-darwin
  lipo -create -output libnewtua_ffi.a \
    target/aarch64-apple-darwin/release/libnewtua_ffi.a \
    target/x86_64-apple-darwin/release/libnewtua_ffi.a
  ```
  Then either point the package's `-L` at that, or (cleaner) wrap it as an
  **XCFramework** with `newtua.h` and depend on the binary target — this removes
  the build-time cargo dependency entirely. Plan this once the app works; don't
  block initial development on it.

## 12. Extending the engine/ABI (only if the UI genuinely needs it)

If a feature needs something the C ABI doesn't expose (e.g. a "keep macOS
metadata" toggle, a streaming read, or surfacing per-failure paths in
`ExtractReport`):

1. Add it in `crates/newtua-ffi/src/lib.rs` (mirror an existing `extern "C"` fn:
   null-check args, `catch_unwind`, map errors to `NtuaStatus`, Rust owns all
   allocations).
2. `cargo build -p newtua-ffi` regenerates `include/newtua.h` (the Swift package's
   header is a symlink to it).
3. Surface it in `bindings/swift/Sources/Newtua/Newtua.swift` as a Swift-native
   API; run `cd bindings/swift && swift test`.
4. Keep `newtua-core` free of user-facing strings — return structured data/codes;
   localize in the front-end.

Known ABI gaps as of this writing: no `keep_macos_metadata` flag on
`ntua_extract`; `ExtractReport` reports only counts (`extracted`/`failed`), not
the failed paths; `read`/`extract` are synchronous and fully buffered.

## 14. The generated template is throwaway scaffolding

Xcode created this from the **SwiftData App** template, so the project ships with
`Item.swift` (a `@Model`) and a `ContentView`/`NewTheUnarchiverApp` wired to a
`ModelContainer` + `@Query`. **newtua has no persistence model** — delete
`Item.swift`, drop the `.modelContainer(...)` from the `App`, and strip the
SwiftData `@Query`/`NavigationSplitView` boilerplate from `ContentView` before
building the real UI. (If you later want to remember recent archives or a default
destination, that's `UserDefaults` + security-scoped bookmarks, not SwiftData.)

The bundled `*UITests` target is fine to keep but will need real UI to test;
don't let its template asserts block you early.
